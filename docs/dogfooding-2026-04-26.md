## Detailed findings

### F-1 [P0] Markdown report footer reads "v0.1" instead of "v0.3.0"

**Subcommand**: `vanish scan ... --output-md`
**OS**: Windows 11
**Severity**: P0 — public-facing artifact (user shares markdown report) carries wrong version → looks unmaintained.

**Repro**:
```powershell
node scripts/index.mjs scan --name "Test Smith" --email "test@example.invalid" `
  --phone "+15551234567" --city "Seattle" --state "WA" `
  --output-md docs/dogfood/scan-synthetic.md --no-banner --no-color
tail -2 docs/dogfood/scan-synthetic.md
```

**Expected**: footer reads `*Generated by Vanish Privacy Scanner v0.3.0*` (or omits the version).

**Actual**:
```
---
*Generated by Vanish Privacy Scanner v0.1*
```

**Suggested fix**: the version is hardcoded in `src/scanner/scan-report.mjs`. Either pull from `package.json`'s `version` field at runtime, or replace the hardcoded string with the current `0.3.0` and add a test in `tests/share-card.test.mjs` or `tests/scan-engine.test.mjs` that asserts the footer matches `package.json`.

### F-2 [P1] `queue list` dumps raw JSON, not a table

**Subcommand**: `vanish queue list`
**OS**: All
**Severity**: P1 — UX. End users who run `vanish queue list` after a real opt-out cycle get pages of raw JSON instead of a scannable status table. This is the primary "did my opt-outs land?" check, and it's currently developer-grade.

**Repro**:
```powershell
node scripts/index.mjs queue list
```

**Expected**: a table like
```
KIND          | TARGET          | STATUS                       | RECHECK
broker        | spokeo          | pending-email-verification   | 2026-06-25
broker        | beenverified    | pending-email-verification   | 2026-05-10
ai-platform   | openai-chatgpt  | pending-reverification       | 2026-06-25
```

**Actual**: full `JSON.stringify(state, null, 2)` of `data/queue-state.json`, including audit trail with HMAC signatures, ~50+ lines for a 4-entry queue.

**Suggested fix**: `scripts/queue-cli.mjs` `list` subcommand should render a table by default, with `--json` flag for the current behavior (preserve scriptability).

### F-3 [P1] Coomer / Kemono `abuseContact` is a vague pointer, not an email

**Subcommand**: `vanish takedown --dmca-letter --coomer --kemono ...`
**Severity**: P1 — letters are partly unactionable until user finds the actual contact themselves.

**Repro**: see generated `docs/dogfood/dmca-test.md`. The "Send to:" lines for Coomer and Kemono read:
- `Send to: DMCA via coomer.su/dmca (varies by mirror)`
- `Send to: contact via kemono.su/contact`

**Expected**: a clickable URL or an email address (e.g., `https://coomer.su/dmca` or `dmca@coomer.su`).

**Actual**: a free-text description. User has to manually visit the site, find the contact, and edit the letter.

**Suggested fix**: `src/takedown/takedown-catalog.json` — coomer.abuseContact and kemono.abuseContact entries should be either a URL (for the form-style takedown) or an email. Update test in `tests/takedown.test.mjs` to assert each `abuseContact` matches `^https?://` or `^[^@]+@[^@]+`.

### F-4 [P1] Some test files don't isolate state writes

**Subcommand**: `npm test` (full chain)
**Severity**: P1 — running tests after real use pollutes the user's real `data/queue-state.json` with test-generated audit entries.

**Repro**:
```powershell
# Run a real opt-out
node scripts/index.mjs opt-out --broker spokeo --email "..." --no-open
# Then run tests
npm test
# Inspect data/queue-state.json — you'll see synthetic test events mixed with your real entries
node -e "const s=JSON.parse(require('fs').readFileSync('data/queue-state.json'));console.log(s.audit.length)"
```

**Expected**: `npm test` should not modify the user's `data/queue-state.json`.

**Actual**: After running our dogfooding sequence + `npm test`, `data/queue-state.json` had 10 audit events (only 2 came from our actual CLI invocation; 8 came from tests).

Affected events seen: `submit_error`, `completed`, `ai_opt_out_submitted_by_user` (timestamps from npm-test runtime, not user runtime).

**Suggested fix**: audit which test files use `createDefaultStore` without overriding `filePath`, and require all integration tests to pass `--state-file <tmpDir>/queue-state.json`. Files known to be clean: `tests/ai-opt-out.test.mjs`, `tests/face-scan.test.mjs`, `tests/clean-ai-history.test.mjs`, `tests/verify.test.mjs` (all use `mkdtempSync`).

### F-5 [P2] HMAC-key warning fires per-mutation, not per-process

**Subcommand**: All commands that persist audit
**Severity**: P2 — output noise.

**Repro**:
```powershell
node scripts/index.mjs ai-opt-out --linkedin --chatgpt --cursor --no-open
```

**Actual**: prints `[vanish] WARNING: Using default audit HMAC key. Set VANISH_AUDIT_HMAC_KEY for trustworthy signatures.` three times (once per follow-up mutation).

**Suggested fix**: in `src/audit/signature.mjs::resolveAuditSecret`, gate the warning behind a once-per-process flag (e.g., a module-level `let warned = false`).

---

### F-6 [P2] `clean-ai-history` size totals are depth-1 only

**Subcommand**: `vanish clean-ai-history --cursor`
**Severity**: P2 — reporting accuracy.

**Repro**: with Cursor installed but mostly-empty:
```
Total local cache: 0 B across 3 path(s)
  ✓ ...\Cursor\User\History (0 B, 3 items)
  ✓ ...\Cursor\User\workspaceStorage (0 B, 2 items)
  ✓ ...\Cursor\logs (0 B, 10 items)
```

User sees "0 B with 10 items" → reasonably confused.

**Suggested fix**: in `src/ai-history/history-engine.mjs::statPath`, when encountering a subdirectory during the depth-1 walk, recurse to compute its byte size (or note "directory size unknown" in the output instead of "0 B"). Cap recursion to ~3 levels to avoid pathological large trees.

### F-7 [P2] BeenVerified opt-out flow asks for `state` but `--state` is not a flag

**Subcommand**: `vanish opt-out --broker beenverified --full-name "Test Smith"`
**Severity**: P2 — minor UX gap; user has to manually fill `state` field on the page.

**Repro**: see B.2 output above. The walkthrough lists:
```
Step 3: Fill the form with:
  ○ State (you need to provide)
```

But there's no documented CLI flag to pre-populate the state.

**Suggested fix**: `scripts/opt-out.mjs` already accepts `--state`. The catalog entry for BeenVerified declares the field but the prompt says "you need to provide" instead of using the value the user passed. Looks like a field-name mismatch (catalog uses `state`, code reads `args.state`?). Verify in `src/adapters/brokers/config/broker-catalog.json::beenverified.optOutFlow.fields`.
